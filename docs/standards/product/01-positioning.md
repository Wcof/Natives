# 产品架构 01 · 定位与边界

> **版本**: 1.0.0 · **日期**: 2026-06-15
> **关联 ADR**: [ADR-0007](../../adr/0007-domain-wheel-reinvention-clarification.md)（不造轮子的适用范围）
> **关联源文件**: `CLAUDE.md`（设计哲学）、`docs/PRD-v2.md`（Solution 段）、`CONTEXT.md`

---

## 一、本篇要约束什么

Natives 最大的产品风险是**边界漂移**：把「基座容器」做成「又一个大而全的应用」。本篇用规则把「什么做、什么不做、由谁来做」钉死，避免后续功能蔓延时反复争论。

---

## 二、产品定位（约束陈述）

**Natives 是「AI Steam Base」——一个桌面基座容器，不是单体应用。** 它的价值在于把现有的 AI 工具、终端、文件管理、插件**粘合**在一个统一外壳下，而非重新实现这些工具。

#### R-P1 · 三种角色与各自的边界
- **等级**：MUST
- **分类**：分层、命名
- **规则**：任何功能在立项时**必须**先归入下表三种角色之一，并按角色对应的约束实现：

| 角色 | 是什么 | 实现约束 |
|------|--------|---------|
| **基座（Host）** | 容器本身：Shell 布局、iframe 管理、Bridge API、终端、环境注入、DB、文件浏览器、AI 工作台 | **必须自建**。没有现成替代品，这是 Natives 的核心价值。 |
| **内置功能页** | 基座自带的页面：首页/商店/工坊/模块管理/设置/文件浏览器/AI 工作台/工具 | **必须自建**，作为基座能力的一部分，**不**通过插件机制加载。 |
| **插件（Plugin/Module）** | 用户安装的第三方页面，运行在 iframe 沙箱 | **必须嵌入现有工具，不造领域轮子**。AI 客户端、代码编辑器等一律嵌入现成方案，不自行实现。 |

- **为什么**：ADR-0007 澄清了「完全不写」只适用于插件层；基座层若不自建，Natives 就退化成普通 Tauri v2（formerly Electron）壳。把角色边界钉死，能阻止「为了省事在基座里塞个简易 AI 客户端」这类诱惑。
- **检查方法**：新功能 PR 描述中**必须**声明它属于哪个角色；reviewer 据此核验实现方式是否匹配。

#### R-P2 · 「不造轮子」只适用于插件层
- **等级**：MUST
- **分类**：分层
- **规则**：
  - **插件层**：禁止自行实现已有成熟方案的领域功能（AI 对话、代码编辑、Git GUI、Markdown 渲染器等）。**必须**嵌入或调用现成工具。
  - **基座层**：禁止把「不造轮子」当作偷懒理由拒绝自建容器基础设施（iframe 容器、Bridge、环境注入、DB 总线）。
- **正例**：插件需要 AI 能力 → 通过终端注入 Claude Code CLI；基座需要主题切换 → 自建 theme-engine。
- **反例**：基座要预览 Markdown → 为了「不造轮子」而拒绝做，让用户自己去装插件 → 错误，文件预览是基座内置功能，必须自建（可基于现成库如 Milkdown）。
- **为什么**：见 ADR-0007。混淆这两层是本项目最常见的产品决策错误。
- **检查方法**：任何「我们不做 X，因为不造轮子」的论述，**必须**明确 X 属于基座层还是插件层。

## 三、能力边界：基座内置 vs 插件

为避免「这个功能该不该内置」的反复争论，下表给出**当前生效**的边界划分。改变边界属于大改动，需走 ADR（见 `README.md` 变更流程）。

#### R-P3 · 基座内置能力清单（封闭集合）
- **等级**：MUST
- **分类**：分层
- **规则**：基座内置功能**必须**限定在以下集合内。新增内置功能需先扩本表并补 ADR：

| 域 | 内置能力 |
|----|---------|
| 外壳 | 三栏布局、侧边栏、命令面板（Cmd+K）、通知中心、设置页 |
| 插件运行时 | iframe 容器、Bridge API、模块安装/管理、权限、心跳/崩溃检测 |
| 终端 | PTY 终端、多会话、环境注入、跟随模式 |
| 文件 | 文件浏览器、文件预览（MD/HTML/代码/多媒体）、文件搜索、Git 只读面板、磁盘用量 |
| AI 工作台 | Agent 变更监控、跟随模式、变更收件箱、会话回放、项目记忆、Skills X-Ray、用量追踪、AI 文件整理 |
| 工具 | 发布向导、截图快递、图片标注 |

- **为什么**：内置能力=基座体积与维护成本。封闭集合迫使每次膨胀都经过显式决策。
- **检查方法**：新功能 PR 若不在表内又想内置，**必须**先开 ADR 扩表。

#### R-P4 · 用户主权原则
- **等级**：SHOULD
- **分类**：交互、主题
- **规则**：基座**应该**把「审美与布局的最终决定权」交给用户。具体表现为：主题可换、侧栏/面板/终端可折叠可调宽、快捷键遵循系统习惯。基座提供容器与连接标准，不替用户做审美裁决。
- **为什么**：这是 Natives 的核心设计哲学之一（CLAUDE.md「User Sovereignty」）。违背它会让 Natives 沦为又一个固定形态的应用。
- **检查方法**：新增「不可配置的视觉/布局硬编码」时，自问是否侵犯了用户主权；若是，提供配置项或交由主题令牌决定。

## 四、明确不做（Out of Scope）

把「不做」写进规范，比写「要做」更能防止蔓延。

#### R-P5 · 基座不做以下事
- **等级**：MUST
- **分类**：分层
- **规则**：基座**必须不**：
  1. 自行实现 AI 对话/推理客户端（交由插件或终端里的 CLI）。
  2. 自行实现代码编辑器（文件预览的轻量编辑除外，深度编辑交由插件）。
  3. 成为跨设备的云同步服务（数据存储仅限本地 `~/.natives/`）。
  4. 自动联网更新已安装插件（插件更新由用户手动触发，见 `CONTEXT.md`）。
  5. 直接执行插件传来的任意代码（插件只能通过声明权限的 Bridge API 间接操作）。
- **为什么**：每一条都对应一次历史诱惑；写下来防止复发。
- **检查方法**：PR 若触及上述任一项，需 ADR 说明为何破例。

## 五、本篇合规自检清单

- [ ] 我的新功能已归入「基座 / 内置功能页 / 插件」三者之一，实现方式与角色匹配。
- [ ] 我没有在插件层重新造已有的领域轮子，也没有在基座层拿「不造轮子」当偷懒借口。
- [ ] 若新增了内置功能，已在本篇 R-P3 表格登记并（如需）开了 ADR。
- [ ] 我没有让基座去做 R-P5 列出的「不做」事项。
