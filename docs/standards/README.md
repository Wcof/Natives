# Natives 规范体系（Standards）

> **本目录是 Natives 项目所有「约束」的唯一权威来源。**
> 当本目录与其它文档（`PRD.md` / `PRD-v2.md` / `architecture/` / `STYLE_GUIDE_AUDIT.md` / `CLAUDE.md` 等）冲突时，**以本目录为准**。
> 其它文档仅作为「现状描述」或「历史决策记录」保留。

---

## 为什么需要这套规范

Natives 已积累了 PRD（两版）、ARCHITECTURE、7 个 ADR、STYLE_GUIDE_AUDIT、4 个 REMEDIATION 计划。这些文档里**约束**与**现状描述**混杂，后续开发者（含 AI 协作者）分不清「必须遵守什么」与「现在恰好是这样」。

本目录把所有**约束**抽离、统一治理：

- **对人**：一眼看清红线，减少争论「该不该这么写」。
- **对 AI**：每条规则可被准确遵守与审查，降低协作偏航。

## 如何使用本规范

### 给人类协作者

1. 动手前：用文档地图定位与你工作相关的 1-3 篇，通读。
2. 动手中：遇到「能不能这么写」的判断，优先查规范，而非凭直觉。
3. 提交前：对照文末「合规自检清单」（本文件底部）。

### 给 AI 协作者（Claude Code / Codex 等）

- **每次接到任务，先加载本 README 与任务相关的规范篇**，再开始编码。
- 规范中的每条规则用统一的**标签 + 四段式结构**书写（见 `00-glossary.md`），AI 应按结构解析、按标签定级。
- 若任务必然违反某条 **MUST** 规则，**停止并向用户确认**，或先补一条 ADR 说明豁免理由，再实施。

## 规则的强度标签（RFC 2119 精简版）

| 标签 | 含义 | 违反后果 |
|------|------|----------|
| **MUST** / 必须不 | 绝对约束，红线 | 禁止合入；违反即视为缺陷 |
| **SHOULD** / 应该 | 强约定，有合理例外 | 需在 PR/ADR 中说明偏离理由 |
| **MAY** / 可以 | 建议、最佳实践 | 鼓励但不强制 |

详细语义与每条规则的归属分类见 `00-glossary.md`。

## 每条规则的书写格式

为了让规则可被人和 AI 一致解析，每条规则按以下结构书写：

```
#### R-[序号] [规则标题]
- **等级**：MUST | SHOULD | MAY
- **分类**：安全 | 数据 | i18n | 无假数据 | 命名 | 性能 | 可访问性 | ...
- **规则**：[一句话陈述约束]
- **正例**：[符合约束的最小代码/做法]
- **反例**：[违反约束的典型写法，及为何有害]
- **为什么**：[约束背后的根本原因]
- **检查方法**：[人/AI 如何验证是否合规]
```

不是所有规则都需要四段全填；**等级 + 规则 + 为什么**是每条规则的最低要求。正例/反例/检查方法在有歧义时必填。

## 变更流程（如何修改本规范）

1. **小改动**（措辞、补充正反例、新增 SHOULD/MAY）：直接修改并随业务 PR 提交，PR 描述注明 `docs(standards): ...`。
2. **大改动**（新增 MUST、废除旧 MUST、改变架构边界）：
   - 先在 `docs/adr/` 写一条新 ADR，说明动机与替代方案。
   - ADR 引用或被引用的规范篇在头部「关联 ADR」处登记。
   - 规范篇正文随 ADR 一并更新。
3. **不允许**：在没有 ADR 的情况下静默放宽 MUST 红线。
