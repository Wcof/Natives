# 前端架构 03 · i18n 双语同步

> **版本**: 1.0.0 · **日期**: 2026-06-15
> **关联 ADR**: 无
> **关联源文件**: `src/i18n/zh.ts`（574 行）、`src/i18n/en.ts`（583 行）、各组件中对 `t()` 的调用

---

## 一、本篇要约束什么

Natives 同时支持中英文（`zh` / `en`），默认 `zh`。i18n 不同步是最常见、最容易被用户发现的低级缺陷——某个按钮加了中文却忘了英文，英文用户看到的就是「undefined」或中文残留。本篇把双语同步钉成红线。

> 这是 `CLAUDE.md`「i18n sync」核心约束的前端落点。

---

## 二、文案键管理

#### R-I1 · 用户可见文案必须走 i18n，禁止硬编码
- **等级**：MUST
- **分类**：i18n
- **规则**：任何**用户在界面上能看到的自然语言文本**（按钮、标签、占位符、toast、空态、错误提示、aria-label）**必须**通过 `t('key.path')` 取值，**禁止**在 JSX 里写死中英文字符串。
- **正例**：`<button>{t('sidebar.collapse')}</button>`
- **反例**：`<button>折叠侧边栏</button>` → 英文环境下仍是中文。
- **例外**：品牌名「Natives」、专有名词（如「Claude Code」）、纯数字/符号可不走 i18n。
- **为什么**：硬编码是 i18n 缺陷的根源；走 `t()` 是唯一可持续的同步方式。
- **检查方法**：review JSX 文本节点是否出现裸中文/英文句子。

#### R-I2 · 文案键按域嵌套，命名用语义而非文案
- **等级**：SHOULD
- **分类**：i18n、命名
- **规则**：文案键**应该**按域嵌套（`nav.*`、`sidebar.*`、`dashboard.*`、`fileBrowser.*`），键名**应该**描述语义而非照抄文案。
- **正例**：`sidebar.collapse`（语义：折叠侧栏）。
- **反例**：`sidebar.foldButton` 照抄某次文案、或扁平的 `text1` `text2`。
- **为什么**：语义键在文案改动时不需要改键名，减少 churn。
- **检查方法**：新增键时核对归属域与命名。

---

## 三、双语同步（红线）

#### R-I3 · zh.ts 与 en.ts 的键必须完全一致
- **等级**：MUST
- **分类**：i18n
- **规则**：`src/i18n/zh.ts` 与 `src/i18n/en.ts` **必须**保持**完全相同的键结构**。新增/删除/重命名一个键，**必须**同时改两个文件。**禁止**只改一边就提交。
- **为什么**：键不一致会导致某一语言下显示 `missing.key` 或 fallback 到另一语言，是用户可感知的缺陷。这是 `CLAUDE.md` 明列的核心约束。
- **检查方法**：
  - 提交前人工核对两边键一致。
  - 推荐：写一个临时脚本对两个文件做深度 key diff（本次规范不强制引入，但鼓励）。
  - `npm run typecheck` 会捕获类型层面问题，但不捕获键缺失——需人工/脚本补位。

#### R-I4 · 新增文案必须同时填中英两语
- **等级**：MUST
- **分类**：i18n
- **规则**：新增文案键时**必须**同时填写中文与英文值，**禁止**用占位（如英文先填中文、或 `TODO`、或复制中文当英文）。若暂时无法提供地道英文，**应该**先填一个合理的直译并标注待润色，而非留空。
- **为什么**：占位会被遗忘；「先中文后补英文」的债务长期不还。
- **检查方法**：新增键时两个值都填；review 时核对英文不是中文原文。

---

## 四、默认 locale 与切换

#### R-I5 · 默认 locale 为 zh，切换即时生效
- **等级**：SHOULD
- **分类**：i18n、状态
- **规则**：应用默认 locale 为 `zh`。切换 locale 后**应该**无需刷新即生效（重新渲染），并持久化到设置。`t()` **应该**响应 locale 变化。
- **为什么**：符合 R-E11（广播同步）与用户主权（产品 01 R-P4）。
- **检查方法**：切换语言后所有可见文本更新且重启后保留。
