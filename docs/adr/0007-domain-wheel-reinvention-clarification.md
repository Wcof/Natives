# ADR-0007: "不造轮子"哲学的适用范围

- **状态**: 已接受
- **日期**: 2026-06-14
- **决策者**: 产品方 + AI 架构师
- **关联**: Q4, Q8

## 上下文

CLAUDE.md 中的"完全不写"哲学说："不造轮子，直接调用或嵌入官方工具（CLI 层环境注入，APP 层窗口嵌入）"。但 PRD 自建了终端、HTTP 服务器、数据库、Bridge 系统等基础设施，看似矛盾。

## 决策

"完全不写"的适用范围是**插件层**，而非基座层。

| 层级 | 哲学 | 实践 |
|------|------|------|
| **插件层** | 不造领域轮子 | 不自己写 AI 客户端、代码编辑器、终端工具，直接嵌入 Claude Code、VS Code 等现有工具 |
| **基座层** | 必须自建 | 容器本身就是轮子，没有现成的"iframe 容户端 + Bridge API + 环境注入"可嵌入 |

## 理由

1. **基座层无替代品** — 市面上没有"通用 iframe 容器 + 插件通信 + 环境管理"的现成方案
2. **插件层有大量工具** — Claude Code、Aider、Cursor 等 AI 工具已成熟，不需要重新实现
3. **基座是粘合剂** — 基座的价值在于把现有工具粘合在一起，而不是替代它们

## 具体实践

### 插件层（不造轮子）
- AI 能力 → 嵌入 Claude Code CLI（终端层环境注入）
- 代码编辑 → 嵌入 VS Code Web 或 Monaco Editor
- 文件预览 → 嵌入现有预览工具
- Git 操作 → 嵌入 Git CLI

### 基座层（必须自建）
- 插件容器 → iframe 管理 + 生命周期
- 插件通信 → Bridge API（postMessage + HTTP）
- 环境管理 → 多组配置 + 加密存储
- 数据持久化 → SQLite + module_data

## 影响

- **CONTEXT.md**：更新"完全不写"术语定义
- **插件开发文档**：强调优先使用现有工具，不鼓励从零实现
